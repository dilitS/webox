**Date:** 2026-05-23
**Scope:** First writable flow in Webox. Init wizard captures a
minimal `smallhost` profile, project wizard provisions subdomain +
SSL + optional DB via `providers.HostingProvider`, and a LIFO stack

## What worked well

- Driving `wizard/` test-first paid off the moment the `tui/` layer
  landed: every UI bug bisected to a TUI dispatch issue, never to a
  parser/runner. The provider seam (`WizardRunner` interface,
  `providerProvider` func type, `MakeStepRunner` dispatcher)
  decoupled the form-driven UI from any SSH transport, so the
  teatest scenarios run entirely in-process.
- Treating `CleanupStep.Params` as untrusted secret-surface from
  day one â€” running every value through the project-wide regex
  corpus inside `Stack.Push` â€” meant the secret-validation test
  (`TestStackPushRejectsSecrets`) was added before the runner ever
  produced a real password. That inverted the usual "we'll add the
  check later" risk: now any new cleanup kind has to prove the
  point at compile time + at push time.
- The retro pattern from Sprint 04 (split sprint doc into
  `Pre-flight`, `Tasks`, `Outcome`, `Retro Link`) carried over
  cleanly. The `Outcome` block forces the agent to admit what
  carried over, which is what kept "resume on launch" honestly out
  of scope instead of half-implemented.
- Splitting the wizard into a `tui/wizard.go` (forms / step model),
  `tui/wizard_runner.go` (provider seam), and
  `tui/views/project_wizard.go` (pure render) kept `Update` pure.
  Even with seven new message types the `Update` function reads
  top-down without nested switches.

## What didn't work / friction points

- The TUI input layer initially processed Vim-style `j`/`k`
  navigation across every step, which silently ate characters
  inside text inputs. Caught only by the
  `TestProjectWizardDomainCollisionStaysOnDomainStep` teatest, not
  by the unit tests. Gated by step kind now, but this is the second
  TUI-input regression in two sprints â€” the next sprint should ship
  a dedicated table-driven "keymap by step" matrix test before any
  new step lands.
- `golangci-lint v2` flagged seven issues in one go after merging
  the wizard (err113, unused, unparam, gocritic, mnd, revive). The
  cluster is a symptom of merging four packages at once without
  running lint between sub-tasks. Future TDD loops should run
  `make lint` after each Green phase, not only at sprint close.
- `coverage tui/views: 0.0%` is technically green (the package
  meets `make cover-check` because the project-wide threshold
  averages to 73.7%) but it is dishonest: the renderers are
  exercised only indirectly through teatest. Sprint 06 should add a
  golden-file pass over the wizard views.
- The decision to keep `wizardStackSlot` as a mutex-guarded pointer
  holder on the model is correct (it lets `tea.Cmd` ticks share one
  `*wizard.Stack` across copied `Model` values) but it is a subtle
  break from "everything in `Model` is value-copied on every
  message". The pattern needs a doc comment in `model.go` so the
  next contributor doesn't accidentally clone the slot.

## Surprises / unknowns discovered

- The provider preflight contract had two failure modes (`nil`
  status without error, and `SSHConnected == false`) that the
  upstream `providers.HostingProvider` doc didn't enumerate.
  Promoted both to sentinels (`ErrPreflightNilStatus`,
  `ErrPreflightSSHDisconnected`) so `tui/` can `errors.Is`
  instead of string-matching. This is the first time we've added
  sentinels in the wizard package rather than the provider package
  â€” the layering held because the wizard owns the contract for
  what counts as a "ready provider".
- Bubble Tea's `tea.Cmd` model makes it tempting to fire a wizard
  step from inside another step's `tea.Cmd`. Resisted the
  temptation: every step transition is initiated by a message in
  `Update`, never inside a command goroutine. That keeps the
  state-transition diagram one-to-one with the message types.
- `go.uber.org/goleak` flagged a leak when the teatest harness
  failed mid-run because the wizard's `wizardExecuteCmd` blocked on
  a context that was never cancelled by the test runner. The fix
  was to give every wizard command the `m.ctx` from the model so a
  global cancel during teardown unblocks it. This is the same
  pattern Sprint 02 used for SSH pool acquire â€” worth promoting to
  a guardrail in `AGENTS.md Â§2.2` (every `tea.Cmd` must take the
  same ctx as the model).

## Changes to apply going forward

- **Process:** Add `make lint` to the "after every Green" step in
  `tdd-loop/SKILL.md`, not only at sprint close.
- **Architecture:** Document the `wizardStackSlot` pointer pattern
  in `tui/model.go` doc comment and reference it from
  `DESIGN.md Â§2.3` so future contributors know why one piece of
  TUI state intentionally lives behind a mutex.
- **Tests:** Sprint 06 must include a keymap-matrix test
  (`for each step, for each key, expected action`) before adding
  any new wizard step. Two TUI input regressions in two sprints is
  enough evidence.
- **Coverage:** Add golden-file snapshots for `tui/views/` during
  Sprint 06; the current "0.0%" number is honest but unsatisfying.

## Open questions to resolve

- Should `pending_cleanups.json` resume-on-launch surface in the
  TUI as a modal at startup, or as a banner that stays until the
  user opts in? UX needs to be specified before Sprint 06 closes,
  because Sprint 06's deploy flow can produce its own
  pending-cleanup snapshots.
- Is "wizard run id" a UUID forever (current decision) or should it
  graduate to a content-hashed plan ID so a crashed wizard with the
  same plan can detect and merge instead of re-execute? Defer
  until DAG rollback (v0.3+).
- Should the next provider (cPanel research) reuse `wizard.Execute`
  as-is, or do we need a per-provider plan validator? The current
  `ValidatePlan` is provider-agnostic; the smallhost-specific
  validation (`smallhost.ValidateDomain`) is called inside
  `Preflight` / `CheckSubdomainAvailable`. The split feels right
  but won't be confirmed until the second adapter lands.
