# Retro: Sprint 12 — Responsive Cockpit Polish

**Date:** 2026-05-24
**Scope:** Re-scoped Sprint 12 from RC1 release prep to responsive TUI ergonomics. Implemented viewport scroll for overflowing frames, rebuilt the Ultra cockpit into a true `2x2 + logs` grid, added `Connections:` fallback for Standard Cockpit, unified cross-screen chrome/titles, refreshed TUI snapshots, and ran `make ci` green.

## What worked well

- Writing the two most failure-prone tests first paid off immediately: one layout test forced `Live Service Topology` out of the old full-width row under logs, and one viewport-scroll test forced the model to gain explicit scroll state instead of hand-wavy "the terminal should just handle it".
- Keeping the scroll contract on `PgUp` / `PgDn` / `Home` / `End` preserved existing `↑/↓` semantics for dashboard selection, live logs, and the CI/CD modal. That avoided a classic TUI regression where one new affordance silently breaks three old ones.
- Reusing `buildTopologySnapshot` for the Standard Cockpit `Connections:` block prevented semantic drift. We now have one topology source of truth that feeds both the ASCII graph and the compact textual fallback.
- Snapshot regeneration stayed cheap because the renderers remain pure. Once the layout and titles were correct, the review artefacts updated cleanly without needing any live backend.

## What didn't work / friction points

- The original Sprint 12 plan mixed two very different categories of work: release hardening (`rc1`, signing, dry-run, GA prep) and terminal ergonomics (overflow scroll, responsive widths, chrome parity). That looked ambitious on paper, but it was the wrong abstraction boundary. The moment new UI feedback landed, the whole sprint had to be re-scoped.
- The first Bento rewrite only moved topology into the left column; it did **not** equalize row heights. Visually that meant CI/CD started one line earlier than topology, so the layout was technically "left of pipeline" but still not a real grid. This was only obvious after reading the failing test output line by line.
- `make ci` caught two linter issues late (`unnamedResult`, `mnd`) that should have been cleaned during the smaller Green steps. Not a big failure, but still evidence that TUI polish work benefits from more frequent lint passes, just like parser work does.

## Surprises / unknowns discovered

- Alternate screen (`tea.WithAltScreen`) fixes host-terminal scrolling, but it does **not** solve overflowing frame ergonomics by itself. If a rendered frame is taller than the viewport, the app needs its own explicit viewport model or the operator loses access to the clipped content. In other words: "fullscreen" and "scrollable when cramped" are separate concerns.
- A side-by-side grid in terminal UI is not just about width math; height balancing matters just as much. Without row equalization, the eye reads the layout as staggered even if the slots are technically correct.
- Cross-screen styling consistency is less about colors than about grammar: once all secondary screens started using the same bracketed emoji titles as the cockpit tiles, the whole app felt more cohesive with surprisingly little code.

## Changes to apply going forward

- **Process:** Do not combine "release hardening" and "UI ergonomics / interaction-model corrections" in one sprint plan. First stabilize the interaction model, then cut RC. This change is applied immediately by re-scoping Sprint 13 around release hardening and moving post-MVP foundation work back.
- **Tests:** For every future cockpit layout change, add one geometry assertion ("is tile X actually left/right/above/below tile Y?") in addition to snapshot tests. Snapshots alone are too noisy to catch spatial intent quickly.
- **Docs:** Keep `docs/UX.md` updated in the same task that changes layout semantics. Layout drift is subtle and expensive because screenshots may look plausible even when the narrative is outdated.
- **Tooling:** Run `go test ./tui/...` after each TUI Green step before the final `make ci`, especially when title copy or layout helpers change. Renderer coverage is good now, but regressions still surface fastest there.

## Open questions to resolve

- Should viewport scrolling also support mouse wheel events in Sprint 13, or is keyboard-only enough for v0.1? The current contract is functional and safe, but mouse-wheel support could improve discoverability.
- Do we want a tiny in-frame scroll hint (for example a dimmed `PgUp/PgDn` indicator) when content is clipped, or is that unnecessary noise once the docs and help text are clear?
