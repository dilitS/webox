# Retro: Sprint 04 TUI Shell

**Date:** 2026-05-23
**Scope:** Bubble Tea / Lipgloss / Bubbles dependency bootstrap, MVU model,
navigation, read-only dashboard, Project Detail Overview, background refresh
command seam, and Sprint 05 planning.

## What worked well

- The TDD loop caught the important architectural shape early: `Update` is
  message-driven and testable without a running terminal.
- Keeping `tui/views` as pure render helpers made it possible to test the
  dashboard and detail screens without coupling view tests to Bubble Tea
  teardown output.
- The existing `status.GetOrFetchMeta`, `services/httpcheck.ProbeHTTP`, and
  `ProbeTLS` APIs were usable without refactor, which validates Sprint 02's
  cache/probe boundaries.

## What didn't work / friction points

- The sprint wording said `config.Load` should return `os.ErrNotExist`, but
  Sprint 01 deliberately made missing config return `DefaultConfig()`. Changing
  that now would be a regression, so first-run detection moved into the TUI load
  command.
- `teatest.FinalOutput` is not a useful stable screen snapshot after quit; it
  mostly contains terminal teardown sequences. Screen assertions have to capture
  `tm.Output()` while the program is alive.
- The initial snapshot tests are smoke tests, not strong golden tests. That is
  honest for the current stage, but it means layout regressions can still slip
  through until goldens are introduced.

## Surprises / unknowns discovered

- Context7 now emphasizes Charm v2 examples with `charm.land/...` imports,
  while Webox docs and sprint acceptance still specify
  `github.com/charmbracelet/...`. Staying on v1 was the lower-risk choice for
  this sprint, but the mismatch should be resolved before public launch.
- Adding `teatest` by pseudo-version pulls a moving `github.com/charmbracelet/x`
  stack. This is expected for `x/exp`, but it increases dependency review
  pressure.
- `goleak` cannot be naively wrapped around `teatest` because the harness keeps
  an internal signal goroutine. Leak tests need to target our own transitions
  and commands, not the harness.

## Changes to apply going forward

- **Process:** For Sprint 05, treat sprint acceptance text as subordinate to
  already-shipped package contracts unless the sprint explicitly calls for an
  ADR/API change.
- **Tooling:** Add stronger golden-file TUI tests once the wizard screens stop
  changing daily; current substring smoke tests are not enough for release
  hardening.
- **Docs:** Track a future decision on Charm v2 import paths. Do not silently
  migrate from `github.com/charmbracelet/...` to `charm.land/...` inside a
  feature sprint.

## Open questions to resolve

- Should Webox keep Charm v1 through `v0.1`, or schedule a dedicated dependency
  migration sprint once `lipgloss` v2 leaves beta?
- Should first-run detection get a documented helper in `config/`, or remain a
  TUI concern that checks the path before calling `config.Load`?
- How strict should Sprint 05 goldens be while the wizard UI is still in motion?
