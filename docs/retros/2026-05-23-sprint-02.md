# Retro: Sprint 02 — SSH + Status Cache

**Date:** 2026-05-23  
**Scope:** TASK-02.1 through TASK-02.7: SSH sentinels/client config, `testing/sshmock`, `ssh.Pool`, exec/keepalive/reconnect, SWR status cache, invalidation metadata, HTTP/TLS probes.

## What Worked Well

- Keeping `testing/sshmock` on `golang.org/x/crypto/ssh` avoided a new dependency and made pool / exec tests exercise the same transport stack as production.
- TDD paid off for `status.Cache`: stale-return + background-refresh semantics are subtle, and tests caught the edge where checking stale via `GetOrFetch` could accidentally trigger a second refresh.
- Real transport tests for `ssh.Pool` gave useful confidence: reuse, max-slot exhaustion, idle cleanup, and concurrent acquire/release all run under `-race`.
- `govulncheck` did its job: it stayed quiet until code actually called vulnerable `x/crypto/ssh` symbols, then blocked the sprint before a vulnerable SSH layer could merge.

## What Didn't Work / Friction Points

- The first pool implementation had a race in `ReapIdle`: `len(p.hosts)` was read before `p.mu`. This is exactly the kind of bug the Sprint 02 risk watch predicted; `go test -race` caught it locally.
- Lint cleanup was clustered late. `errcheck`, `err113`, `mnd`, `gocritic`, and `gosec` all fired at once after several task commits. For the next sprint, run `make lint` after each task-sized commit, not only before final PR.
- The repo still had Go 1.24 assumptions in contributor docs. Sprint 02 forced a toolchain-floor change to Go 1.25 because the safe `x/crypto` release requires it.

## Surprises / Unknowns Discovered

- `x/crypto v0.45.0` is the last Go 1.24-compatible line, but several reachable SSH vulnerabilities are only fixed in `v0.52.0`, which declares Go 1.25. There was no safe Go 1.24-compatible landing point once Webox started calling SSH client/server symbols.
- `x/crypto/ssh.NewServerConn` is enough for the current mock scope. We do not need `gliderlabs/ssh` until proven otherwise.
- Keepalive is easiest to test by counting global requests server-side, not by trying to observe client internals.

## Changes To Apply Going Forward

- **AGENTS.md:** updated the language/toolchain row to Go 1.25+ and changed the `go get` gotcha: pin to the current floor unless `govulncheck` proves the safe fix requires raising it.
- **Process:** run `make lint` after each task commit in Sprint 03. Waiting until final CI concentrates trivial lint fixes into a noisy commit.
- **Tooling:** keep `govulncheck` in the mandatory local `make ci`; it caught a real reachable issue created by this sprint.
- **Sprint 03:** provider parser work must use real fixtures as soon as live `devil` output is available; until then, fixtures need provenance notes and malicious variants.

## Open Questions To Resolve

- Do we need a small ADR for the Go 1.25 floor bump, or is `CHANGELOG.md` + `AGENTS.md` + this retro enough? Current decision: enough, because the reason is a security patch path rather than an architecture model shift.
- Should `ssh.Pool` expose runtime metrics (active/idle per target) for `webox doctor` later? Defer to post-MVP unless debugging pool leaks becomes painful.
