## Path selection in 90 seconds

The decision matrix in the sprint plan filled itself:

- **Path C (Public Launch)** rejected without conditions: `v0.2.0-rc1` is *currently blocked* on TASK-22.0 (operator-side cpanel credential rotation). Launching with research-derived fixtures would damage the preset registry's verified audit trail at exactly the moment it gets the most scrutiny. Defer until `v0.2.0` GA tag.
- **Path B (CyberPanel)** rejected and deferred to v0.4+: requires root on the host, threat model widens, no demand signal from operators. The threat-model RFC alone is a separate sprint of work; not worth the cost at this gate.
- **Path A (DirectAdmin)** accepted. Three reasons stacked: (1) DA is the 2nd-largest panel in PL by market share (VPS/dedi segment where cPanel licensing is pricier); (2) the first 3 tasks need *no live test account* because DA's Live API is documented via a publicly bundled Swagger spec — the same starting baseline cpanel had after Sprint 21; (3) Path A can ship in parallel to the operator's cpanel rotation without contention.

## What worked well

- **Decision matrix-as-code paid off again.** The pre-written matrix in the Sprint 23 plan (filled at retro per the original spec, but I filled it at sprint-start instead because the inputs were already knowable) collapsed what would have been 30 minutes of "should we…" into a 2-minute write-up with one chosen path and two written-down rationales for the rejected ones. The plan-doc now reads as both decision record and implementation roadmap — single source of truth.
- **Mirror discipline from Sprint 21 / 22 made TASK-23.1 and TASK-23.2 fast.** Reusing the `transport.go` + `client.go` + `ssh.go` + `composite.go` + typed sentinels shape from cpanel/uapi meant I could focus on the DA-specific behaviour (`ErrAPIDisabled`, three wire shapes for the decoder, loopback-curl SSH) rather than relitigating retry policy or error taxonomy. The package went from empty directory to 88.7%-covered read-only client in one TDD cycle.
- **`ErrAPIDisabled` is the right escape hatch for DA's two-API reality.** DA installs disagree about whether `/api/*` (Live API) is enabled. Older installs and security-conscious hosters only expose `/CMD_API_*` (legacy). Instead of trying to handle both transports in Sprint 23, I added a single typed sentinel (`ErrAPIDisabled`) that fires on 404 + 503-with-canonical-body and routes to `DISABLED` status in the doctor CLI. The remediation note ("contact host to enable Live API or wait for a CMD-only adapter in v0.4+") is actionable and the typed boundary makes the future legacy adapter an additive change instead of a refactor.
- **The SSH fallback for DA solved a *different* problem than cpanel's.** cpanel's SSH path exists because the panel has a first-class `uapi` CLI. DA has no equivalent universal CLI for user-level operations, so the fallback shells out to `curl` against `https://localhost:2222/api/...` from inside the box. The use case it handles ("operator can SSH but can't reach :2222 directly") is real and different from cpanel's ("HTTPS rate-limited, shift to SSH"). Writing this out in the package doc (`ssh.go` top comment) ahead of the implementation helped me reject the "let's just port the cpanel SSH layer wholesale" temptation.
- **`splitOnLastNewline` + curl `--write-out '\n%{http_code}'` trick is reusable.** The fallback needs to know the HTTP status because the panel returns useful JSON bodies *with* 401 / 404 — not just an empty error. The `--write-out '\n%{http_code}'` trick appends the status to stdout, and a one-pass byte scan splits it back out. Cheap, deterministic, no parsing-the-output magic. Worth lifting into an `internal/sshcmd` package when a third adapter joins (Sprint 24+).
- **`make ci` zielony pierwsze podejście.** 79.3% total coverage. `make lint` clean (after two `//nolint:dupl` blocks documented). `go test ./providers/directadmin/...` 85.2% coverage. The cpanel/uapi pattern carried over without any drift; the only friction was the parser's gocyclo budget after the new flags were added.

## What didn't work / friction points

- **golangci-lint `dupl` fired on `writeCpanelText` ↔ `writeDirectadminText` and the two native SSH runners.** I expected this — the two doctor CLIs have parallel-but-not-identical structure, and the per-provider typed errors (`uapi.ErrTransportUnavailable` ≠ `daapi.ErrTransportUnavailable`) prevent a clean shared helper. The right fix is an `internal/sshcmd` package that takes a wrap-target as an argument; deferred to Sprint 24+ when a third provider proves the abstraction. For now: two `//nolint:dupl` blocks with explicit "revisit when N≥3" comments documenting the deferred refactor. The honest read is: cpanel and directadmin look like 80% the same code, but the 20% difference is exactly the part that needs to *not* be refactored away — type-system enforcement of "use this provider's sentinel".
- **Parser gocyclo budget needed two helper extractions.** Adding `--api-port`/`--ssh-port`/`--no-ssh`/`--no-api`/`--loginkey` to the parser pushed `applyCpanelPrefixFlag` over the gocyclo budget. The fix (extracting `applyAPIPortFlag` + `applySSHPortFlag` as standalone helpers that switch on `doctorTarget`) is cleaner than the inline-everything original, but each new doctor target costs me a 5-minute refactor. Worth pre-planning a shared `applyPortFlag(parsed, name, portStr, scope)` if Sprint 24+ adds another panel (`cyberpanel` would mean re-extracting again).
- **DA preset schema didn't have `app_root_template`.** I instinctively added the field when graduating the preset to `candidate` because the adapter would need it later. The schema rejected it (`additionalProperties: false`). Two ways forward: (1) extend the schema with the optional field, (2) push the resolution into the adapter's Properties layer (where cpanel does it). I picked (2) for Sprint 23 to keep the schema unchanged and the PR boundary clean; Sprint 24's adapter will own the `app_root_template` resolution. Lesson: when the preset schema says no, push the adapter-specific abstraction one layer deeper instead of widening the schema.
- **`t.Parallel()` × `t.Setenv()` incompatibility flagged me again.** Same bug class Sprint 22 hit on the cpanel E2E suite. The directadmin tests don't `Setenv` so they parallelise cleanly — but the pattern "test with env var → must be serial" is now a Sprint 22 retro action item that hasn't propagated into the gotchas list yet. Filing as a `docs/gotchas.md` update for Sprint 24.

## Surprises / unknowns discovered

- **DA Live API endpoint shape varies more than the Swagger spec suggests.** Three wire shapes documented in [`decodeList[T]`](../../providers/directadmin/api/client.go): modern wrapper (`{"domains":[...], "success":true}`), generic data wrapper (`{"data":[...], "success":true}`), bare top-level array. Researching three shapes ahead of the live capture saved me from a Sprint 24 panic when the live response shape inevitably matches none of them — the decoder's shape-tolerance is the safety net.
- **DA's MySQL user prefix cap (8 chars) is hard-coded in the panel.** Found by searching the JBMC docs for "username prefix". Not in the Live API spec; surfaced as a `known_risks` entry in the preset and an explicit follow-up for Sprint 24's adapter validator. Without enforcement, every `verylongusername_db` provisioning attempt would fail at the panel layer with a useless error.
- **The cpanel doctor's `transportLabel` type assertion pattern doesn't generalise.** I copied it for directadmin's `directadminTransportLabel` and immediately saw the smell: every new transport type requires a new case. If/when a third provider lands, this becomes 3 functions doing the same dispatch. Sprint 24+: consider adding `Transport() string` to the `Reader` interface so each implementation labels itself. Deferred because the interface budget (5 methods) is at the cap.
- **DA's "Live API" terminology is operator-facing.** I almost named the doctor's HTTPS toggle `--no-uapi` (mirror of cpanel). Then I checked DA's docs — they explicitly call their JSON API "Live API", not "UAPI" (cpanel's term). `--no-api` is the right name; it matches what an operator typing `webox doctor directadmin --help` would expect after reading DA's own docs. Naming hygiene saved a future bug report.

## Changes to apply going forward

- **`docs/gotchas.md`**: add row "When a Go test uses `t.Setenv(...)`, the same test cannot call `t.Parallel()` — the test runtime detects the conflict at runtime and panics. The cpanel E2E suite (Sprint 22) and Sprint 23 retro both hit this. Either drop `t.Parallel()` from env-var-mutating tests, or move the env-var dependency into a config parameter the test injects explicitly." — captures the second instance of the pattern.
- **Sprint 24 plan**: scope an `internal/sshcmd` package extraction as the first task. It takes the wrap-target as an argument so each provider package can stay on its own typed `ErrTransportUnavailable`. Two callers today (cpanel runner, directadmin runner); a third provider (CyberPanel? cPanel-DA hybrid? PleskAPI?) tips the value calculation.
- **Sprint 24 plan**: scope a `Reader.Transport() string` method addition (or move the label logic into a per-Reader function table) when the third adapter lands. The current `directadminTransportLabel` is a 4-case type-switch that will become a 6-case switch with one more provider; not worth living with for long.
- **Sprint 24 plan**: live test account procurement for DA is the critical-path blocker for the full adapter. Mirror the cpanel TASK-22.0 motion: operator picks a host (hitme.pl, lh.pl, lub direct-admin.io sandbox), rotates credentials, runs `scripts/smoke-directadmin.sh` (to be written), captures fixtures into `providers/directadmin/api/testdata/live/`, manually redacts, promotes to checked-in fixtures.
- **`AGENTS.md` §1 (Guardrails)**: no change. Sprint 23 stayed inside the existing guardrails (HTTPS-only, typed sentinels, atomic config, no telemetry, pinned GHA SHAs, etc.). The DA adapter inherits all of them cleanly.
