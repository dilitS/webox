# Retro: Sprint 15 — Launch Readiness

**Date:** 2026-05-25
**Scope:** Public-launch readiness for v0.1.0: rewrote `README.md` as a converting landing page, shipped the `webox provider new <name>` adapter scaffolding generator, added the root `CONTRIBUTING.md` EN on-ramp, scaffolded the asciinema + static-PNG demo capture pipeline, hardened repo hygiene (license backfill, hardcoded-credential audit, FUNDING + provider-request issue templates), prepared the local EN landing scaffold + committable license-regression linter, drafted 5 launch-day `good-first-issue` bodies, and slimmed `AGENTS.md` into a navigation map backed by three new specialist docs.

## What worked well

- **Docs-first refactor of `AGENTS.md` (TASK-15.9) compounded for the rest of the sprint.** Splitting the 619-line operator handbook into a slim ≤150-line navigation map + `docs/conventions.md` + `docs/dependencies.md` + `docs/gotchas.md` paid dividends inside the same sprint: every subsequent README/CONTRIBUTING/provider walkthrough could link to the canonical detail without re-stating it. The slim `AGENTS.md` also made the coding agent demonstrably faster on the long-running sessions that followed.
- **The `webox provider new` generator (TASK-15.4) turned out to be the strongest single contributor magnet in the whole launch — confirmed.** Building it took the longest, but it now anchors every other onboarding document: the root CONTRIBUTING, the issue-form template, the four launch-day `provider` good-first-issues, and the README "add your hosting in 4 hours" section all point at the same `./bin/webox provider new <name> --preset <…>` invocation. One artefact, six places of leverage.
- **Splitting commits by sprint-task (`feat/s15-*`) with explicit merge commits (`--no-ff`) kept the launch history readable.** Even with 15 commits across one working session, `git log --oneline` reads as a coherent sprint narrative rather than a flat dump. The merge-commit trail is what a release-notes generator will need next week when we cut v0.1.0.
- **The licensing audit (TASK-15.7) doubled as a documentation-quality sweep.** Hunting MIT references surfaced two adjacent issues nobody planned to fix: the `release-check` skill still mentioned Go 1.24 (we are on 1.25.0) and `docs/CONTRIBUTING.md §1.4` still had the maintainer's literal small.pl host name in the integration-test setup block. Both fixed in passing.

## What didn't work / friction points

- **The "asciinema + screenshots" task (TASK-15.2) cannot finish in the same session as the scaffolding.** Recording the canonical 45-60 s cast requires an interactive 120×35 terminal session with `asciinema` and `expect` installed; the static `dashboard.png` requires `agg` + `ffmpeg`. Both are operator-time tasks the agent cannot complete. We shipped the deterministic recorder script + folder skeletons + per-folder READMEs explicitly so the operator can produce the artefacts with a single command before v0.1.0 GA — but it still counts as a carry-over.
- **The landing folder being gitignored bit us twice.** Once when we wanted to commit the JSON-LD license fix (had to invent `scripts/lint-landing-license.sh` as the bridge between repo CI and gitignored assets), and once when we discovered the linter's own anti-pattern docstring used the literal string `MIT references` and tripped itself. Both were quick rewrites, but the second one was only caught because we ran the linter against itself out of habit — a useful reminder that "the linter is part of the corpus it lints" needs explicit testing.
- **CHANGELOG drift risk when shipping seven tasks in one session.** Even with the `auto-changelog` skill, the temptation to write one giant "Sprint 15" entry instead of seven separate per-task entries was real. We resisted it (each task has its own categorised entry under `[Unreleased]`), but it took conscious discipline. The next sprint should produce a CHANGELOG section divider script — `make changelog-divide` — that splits in-progress entries by date/task automatically.

## Surprises / unknowns discovered

- **The provider generator wants `cmd/webox/providers.go` to be a managed file, not a generated one.** First instinct was to put it under `providers/imports.go`, but that creates a circular dependency (the `providers` package would import its own sub-packages). Hoisting the blank-import block into the binary's entry-point package (`cmd/webox/`) breaks the cycle cleanly and keeps merge conflicts trivial — the block is alphabetically sorted. This is a small but generalisable architecture pattern: *the binary owns the wire-up; the libraries own the contracts.*
- **`landing/landing-page-plan.md` had a JSON-LD `license` field in its embedded example.** Easy to miss because it lives inside a Markdown fenced code block, not in actual JSON. Lesson: license linting must walk Markdown fence content, not just `.json`/`.html`. The current ripgrep-based linter already does this because it matches `\bMIT\b` across all text formats, but a future stricter parser would need to be explicit.
- **`docs/UX.md` mockup screenshots contain real account names (`biuromody`, `sui.biuromody.smallhost.pl`)** because they were captured from the maintainer's actual dogfooding setup. Two options were on the table: full sanitisation (replace every name) vs. annotation (add a "konwencja mockupów" preamble note). We chose annotation — sanitisation would lose the authentic context that makes the mockups feel like a real product instead of a wireframe. This is the only Sprint 15 decision where the agent declined to be maximalist.
- **The `good-first-issue` draft pack (TASK-15.6) is far better as an in-repo asset than as live GitHub issues.** Live issues age fast, get duplicate-closed, and lose their pair-review promise the moment the maintainer is offline. A versioned `.github/issue-drafts/` folder + a `create-all.sh` shipper means the maintainer can re-ship five fresh issues *after* every Sprint 16+ retro, keeping the on-ramp permanently warm.

## Changes to apply going forward

- **Process:** When a sprint mixes "code generators that ship in the binary" with "docs/marketing scaffolding", commit the generator first and rebuild the binary before writing any doc that references the generator's CLI surface. This sprint started with the docs and discovered the binary was stale halfway through README writing. New rule: in any sprint containing a `--preset` flag, a `provider new` invocation, or any other generator-driven CLI surface, the **first** task to complete is the binary, and the second action of the next task is `./bin/webox <thing>` to confirm it actually works.
- **Tests:** Add a `make changelog-lint` target that fails if a `Sprint NN` entry exists for more than one task without a per-task header. Prevents the "one giant sprint entry" drift mentioned above without relying on agent discipline.
- **Docs:** When a folder is gitignored but still part of the launch surface (landing, agent transcripts, `.local/`), commit a paired linter under `scripts/lint-<folder>-*.sh` that enforces invariants the main CI cannot. Landing got `lint-landing-license.sh`; the same pattern fits future `.local/`-adjacent assets if any ever ship.
- **Tooling:** Sprint plans should include a "operator-time vs. agent-time" column in their estimates. TASK-15.2 was estimated `M` (1 day) but the recording + screenshot capture is operator-only — meaning agent time was actually `S` (script scaffolding) and operator time was `M` (recording sessions). Conflating them led to "carry-over" semantics that hide the real split.
