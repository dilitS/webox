# Retro: Sprint 06 — GitHub Deploy Workflow

**Date:** 2026-05-23  
**Scope:** Resume-on-launch, GitHub service client, workflow templates,
GitHub rollback kinds, first-deploy polling helper, keymap/golden coverage,
and Sprint 07 planning.

## What Worked Well

- Existing LIFO stack design was strong enough to absorb GitHub resources
  without changing rollback strategy to DAG.
- Keeping `gh` as primary transport avoided storing user auth in Webox while
  REST fallback stayed possible for headless/PAT flows.
- Existing `x/crypto` dependencies were enough for GitHub sealed-box secrets,
  so Sprint 06 did not need a new dependency just to encrypt Actions secrets.
- Resume-on-launch fit cleanly into MVU: file reads and deletes are `tea.Cmd`,
  `Update` remains pure.

## What Didn't Work / Friction Points

- Sprint 06 was over-scoped for one uninterrupted implementation pass. The
  foundational pieces landed, but full TUI Repo/Secrets/Workflow/Deploy screens
  and live post-deploy verification belong in Sprint 07.
- The sprint spec said `gopkg.in/yaml.v3` was already in `go.mod`; it was not.
  Because AGENTS requires maintainer sign-off for new deps, workflow validation
  had to stay as conservative invariant checks.
- GitHub Actions `${{ ... }}` expressions conflicted with Go template
  delimiters. The workflow renderer now uses `[[ ... ]]`, but this should have
  been called out in the sprint spec.
- Workflow-file rollback is less trivial than "delete path": GitHub Contents
  API requires the current file SHA before DELETE. Tests now encode that.

## Surprises / Unknowns Discovered

- `gh api` plus stdin bodies gives a good middle path: auth stays in gh, but
  code can still target exact REST endpoints.
- Full `doctor github` is more than a quick check because the useful version
  needs config-aware repo metadata and a stable JSON schema.
- `last_deploy` should be injected into status fetching rather than hidden
  behind globals; Sprint 07 should avoid quick package-level seams.

## Changes To Apply Going Forward

- **Process:** split future sprint docs into "foundation", "TUI wiring", and
  "live verification" tracks; Sprint 06 mixed all three and made the done
  boundary too optimistic.
- **Docs:** dependency assumptions in sprint plans must be verified against
  `go.mod` during pre-flight, not discovered during implementation.
- **Tooling:** consider a small workflow-template linter target once YAML
  parser sign-off is resolved.

## Open Questions To Resolve

- Should `gopkg.in/yaml.v3` be approved for workflow AST validation, or should
  Webox keep a no-new-dependency validator for v0.1?
- Should automatic repo creation remain default, or should existing-repo mode
  become default to reduce PAT scope requirements noted in AUDIT B7?
- What exact JSON shape should `webox doctor github --json` expose for
  per-project deploy key and last-run metadata?
